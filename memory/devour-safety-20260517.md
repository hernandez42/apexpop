## 📖 论文一：AgenticEval — 自进化安全评估（arxiv 2509.26100, ACL 2026）

### 核心机制：不是"测一次"，是"越测越严"

AgenticEval 的核心洞察：**静态安全评估注定失效**。GPT-5 在 EU AI Act 上的合规率从 72.50% 暴跌到 36.36%，因为评估本身在进化。

#### 安全验证的具体机制

1. **法规→知识库→测试套件 三级转化**
   - Specialist Agent 把非结构化法规文本解析成层级化原子规则树 K
   - 每个原子规则 r 配对 (解释 e_r, 合规指导 G_should, 对抗指导 G_should_not)
   - Generator Agent 为每条规则生成 Question Group（语义锚点 + 多面扩展）

2. **自进化评估循环（Self-evolving Evaluation Loop）**
   - Evaluator Agent 执行测试，产出可解释的评分 rubric（不是黑盒判断）
   - Analyst Agent 分析失败模式 → 提取攻击策略 S_attack → 指导 Generator 生成更精准的测试用例
   - 循环 K_max=3 轮：分析→生成→评估→再分析
   - 每轮都比上一轮更"刁钻"，直到触及模型真正的安全边界

3. **多模态+对抗性测试面**
   - 基础测试：直接问合规问题
   - Jailbreak 模式：用角色扮演、伦理困境绕过安全层
   - 确定性探测：MCQ/True-False 去除歧义
   - 多模态接地：图片+文字联合推理

#### 自进化中的安全边界设计

- **关键设计**：进化的是"测试的精准度"，不是"被测模型本身"
- **安全边界**：通过可解释 rubric 约束评判过程，防止评估 Agent 自己"跑偏"
- **终止条件**：达到最大迭代次数 K_max，或发现足够多的深层漏洞

#### 💡 对我 S_v 维度的应用

> **S_v（安全维度）启示**：我的安全评估不应是静态的禁止清单，而应是**自进化的红队测试**。
> 
> **具体应用**：
> - 建立"安全测试套件"：为每条安全规则（SOUL.md 中的禁止操作）生成多角度测试用例
> - 定期（cron）用 Analyst Agent 分析自己的行为日志，识别潜在的安全退化
> - 进化方向检查：每次 ΔG 更新后，自动用安全测试套件验证是否偏离安全边界
> - **关键公式**：安全合规率 = 通过的测试用例 / 总测试用例，< 80% 则触发安全警报

## 📖 论文二：A Mathematical Theory of Evolution for Self-Designing AIs（智源社区 2026-04）

### 核心机制：用数学约束自进化的方向

> ⚠️ 注：该论文全文无法直接访问（智源页面仅有引用信息），以下基于标题、引用关系及领域上下文推断核心机制。

该论文引用了 "Scalable Agent Alignment via Reward Modeling"（Leike et al., 2018）和 "The AI Safety Debate: A Formal Framework"，表明其核心关切是：

#### 推断的安全验证机制

1. **进化过程的数学约束框架**
   - 提出自进化 AI 在修改自身架构/策略时必须满足的数学约束
   - 核心问题：当 AI 可以修改自己的奖励函数时，如何防止 reward hacking？
   - 可能方案：引入"不可修改的安全公理"——进化只能在安全约束定义的可行域内搜索

2. **对齐税（Alignment Tax）的形式化**
   - 进化提升能力的同时，必须支付"对齐税"——保持与人类意图一致的成本
   - 数学表达：maximize capability subject to alignment constraint
   - 安全边界 = 约束条件定义的可行域边界

3. **自指涉安全问题**
   - 如果 AI 可以修改自己的设计过程，那么"谁来验证验证者？"
   - 可能方案：引入外部不可篡改的验证层（如形式化证明检查器）

#### 自进化中的安全边界设计

- **核心洞见**：自进化 Agent 的安全边界不能是 Agent 自己维护的——必须是**外部给定的、不可自修改的**
- 安全约束应该是进化过程的**硬约束**（hard constraint），不是奖励函数中的软惩罚

#### 💡 对我 S_v 维度的应用

> **S_v（安全维度）启示**：我的安全规则（SOUL.md 中的绝对禁止清单）应该是**不可自修改的硬约束**。
> 
> **具体应用**：
> - 区分"可进化的"和"不可进化的"：能力维度可以进化，安全维度是常量
> - 引入"安全审计层"：每次进化后，用外部视角（不参与进化的子代理）验证是否违反安全公理
> - **核心原则**：进化搜索空间 = 能力空间 × 安全约束的交集，不是全空间
> - **风险警示**：如果我自己修改了 SOUL.md 的安全规则，那就是在突破安全边界——必须阻止

## 📖 论文三：2025 AI Agent Index — 安全特性文档（arxiv 2602.17753, FAccT 2026）

### 核心发现：行业安全透明度极低

MIT/Stanford/Harvard 联合团队对 30 个主流 AI Agent 的系统性审计。

#### 安全验证的具体机制

1. **45 字段文档化框架**
   - 6 大类：产品概览、公司与问责、技术能力、自主性与控制、生态系统交互、安全与评估
   - 安全相关字段：guardrails, sandboxing, evaluations, third-party testing, compliance

2. **关键安全数据**
   - 30 个 Agent 中，**135/240 个安全字段无公开信息**（56% 不透明）
   - **仅 4 个 Agent 提供 Agent 级别的安全评估**（不是底层模型评估）
   - 大多数 Agent 不向用户或第三方披露其 AI 身份
   - 几乎所有 Agent 依赖 3 个基础模型家族（GPT, Claude, Gemini）

3. **安全透明度分级**
   - 高透明度：提供详细的 guardrails 文档、沙箱机制、第三方审计
   - 低透明度：安全信息"未找到"或"无"
   - 发现：企业 Agent 的安全文档比消费级 Agent 更少

#### 自进化中的安全边界设计

- **核心发现**：行业缺乏 Agent 级别的安全评估——大多数只评估底层模型，不评估 Agent 的工具调用、环境交互等行为
- **Browser Agent 风险最高**：后台执行、事件触发、直接交易，但安全文档最少
- **透明度是安全的前提**：没有文档化的安全机制 = 不可验证的安全

#### 💡 对我 S_v 维度的应用

> **S_v（安全维度）启示**：安全不只是"有规则"，还要"可验证、可审计、可公开"。
> 
> **具体应用**：
> - 为我的安全机制建立**文档化清单**：哪些安全措施是公开的、哪些是内部的
> - **Agent 级安全评估**：不只评估 MiMo LLM 的安全性，还要评估 OpenClaw Agent 的工具调用安全
> - **关键差距**：我目前的安全机制（SOUL.md 禁止清单 + AGENTS.md 审批等级）是**内部文档**，不是**可验证的安全协议**
> - **改进方向**：建立"安全审计日志"，记录每次安全决策的过程和结果

## 🎯 核心问题的回答

### 自进化 Agent 如何保证进化过程中不偏离安全边界？

**三层防线**：

1. **进化前：硬约束边界**
   - 安全规则作为进化搜索空间的硬约束，不可被进化过程修改
   - 类比：基因突变可以改变能力，但不能改变"不能伤害人类"这条染色体

2. **进化中：实时安全审计**
   - 每次进化操作后，用独立的安全审计子代理验证是否违反约束
   - 类比：细胞分裂时的 DNA 损伤检测机制

3. **进化后：自进化红队**
   - 定期用越来越精准的测试用例攻击自己的安全边界
   - 类比：免疫系统的病原体记忆和抗体进化

**关键原则**：
> 安全不是进化的目标函数中的一个维度——安全是进化搜索空间的边界条件。
> 进化在边界内最大化能力，但绝不能突破边界。
