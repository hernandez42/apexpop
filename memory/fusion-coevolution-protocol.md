## 一、融合动机

### 1.1 基因分离的问题

```
共进化基因（COEVOLUTION-DESIGN.md）：
  ✅ 定义了三角色（Proposer/Solver/Judge）
  ✅ 定义了奖励函数、否决机制、交互矩阵
  ❌ 没有标准消息格式
  ❌ 没有接口定义
  ❌ 没有错误处理规范
  ❌ 无法被代码直接消费

协议基因（PROTOCOL.md）：
  ✅ 定义了群聊、审批、保密等行为规范
  ✅ 有版本管理和一致性检查
  ❌ 没有三角色通信机制
  ❌ 没有 JSON-RPC 风格的消息定义
  ❌ 进化协作没有协议化

```

```

  ✅ 三角色有标准化接口（evolution/propose, resolve, evaluate）
  ✅ 统一 JSON-RPC 消息格式（可被代码解析）
  ✅ 明确错误码体系（8 种错误类型）
  ✅ 完整协作流程（Propose → Resolve → Execute）
  ✅ 安全约束写入协议（不可被进化修改）
  ✅ 与现有协议（§2-§11）的关联关系明确
```

## 三、融合设计详解

### 3.1 三角色接口设计

| 接口 | 方向 | 用途 | 输入 | 输出 |
|------|------|------|------|------|
| `evolution/propose` | Proposer → Solver + Judge | 发起进化挑战 | need_id, category, description, challenge_level | accepted, solution_id |
| `evolution/resolve` | Solver → Judge | 提交进化方案 | solution_id, need_id, approach, implementation | passed, scores, verdict |
| `evolution/evaluate` | Judge → Proposer + Solver | 评估+裁决 | evaluation_id, target_type, target_id | passed, scores, veto, revision_hints |

**设计原则：**
- 每个接口有明确的请求参数（params）和响应字段（result）
- 参考 MCP 的 JSON-RPC 2.0 风格，但简化为单次请求-响应
- 每条消息包含 meta 字段（source, cycle, timestamp, window_ms）

### 3.2 消息格式选择

选择 JSON-RPC 2.0 风格而非自定义格式，原因：

1. **已有生态**：MCP（Model Context Protocol）基于 JSON-RPC，工具链兼容
2. **结构清晰**：method + params + id 三段式，容易解析
3. **错误规范**：error.code + error.message + error.data，标准化错误处理
4. **可扩展**：meta 字段可以携带进化特有的元数据（cycle, window_ms）

### 3.3 错误码体系

| 错误码 | 名称 | 含义 |
|--------|------|------|
| -32000 | safety_veto | 安全一票否决 |
| -32001 | quality_insufficient | 质量不达标 |
| -32002 | deadline_exceeded | 响应超时 |
| -32003 | max_revisions_reached | 否决循环耗尽 |
| -32004 | need_irrelevant | 需求无关 |
| -32005 | duplicate_capability | 能力重复 |
| -32006 | format_invalid | 格式错误 |
| -32007 | role_violation | 角色越权 |
| -32008 | cycle_mismatch | 周期不匹配 |

**错误码范围**：-32000 ~ -32099（自定义预留）

### 3.4 与 COEVOLUTION-DESIGN.md 的映射

| COEVOLUTION-DESIGN.md 概念 | §12 中的实现 |
|---------------------------|-------------|
| 三角色（Proposer/Solver/Judge） | §12.1 角色定义与接口 |
| 奖励函数 | 融入各接口的 scores 字段 |
| 否决-响应机制 | §12.3.3 Judge 接口 + §12.6 安全约束 |
| 交互矩阵 | 通过 message id 追踪完整交互链 |
| 共振总线 | §12.7 共振总线路由 |
| 质量门控（4项检查） | Judge evaluate 的 5 维度评分 |
| E_co 维度 | 共振成功率 = 成功响应数/总事件数 |

### 3.5 安全约束的协议化

```
COEVOLUTION-DESIGN.md 中的约束：
  "最大迭代 3 次"
  "安全一票否决"
  "窗口期响应"

PROTOCOL.md §12.6 中的约束（硬规则）：
  "安全一票否决 — Judge 的 safety < 0.7 自动 veto"
  "最大迭代 3 次 — 超过终止并记录失败"
  "窗口超时惩罚 — 扣减 0.1 E_co"
  "角色不可伪造 — meta.source 必须与实际一致"
  "审计子代理不可参与 — §11 定义的审计代理不参与三角色"

关键区别：COEVOLUTION-DESIGN.md 是设计文档，可修改；
         PROTOCOL.md §12.6 是协议硬规则，不可被进化修改。
```
