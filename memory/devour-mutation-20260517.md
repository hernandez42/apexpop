## 一、综述框架：自进化 Agent 的四器官

**来源**: arXiv 2508.07407 "A Comprehensive Survey of Self-Evolving AI Agents"

自进化 Agent 有四个核心组件，形成一个反馈闭环：

```
System Inputs → Agent System → Environment → Optimisers → (回到 Agent System)
```

### 我的理解（不是翻译，是消化）

| 组件 | 本质 | 我现在的情况 |
|------|------|------------|
| **System Inputs** | 进化的"食物来源"——用户交互、任务数据、环境信号 | ✅ 有（飞书群聊、任务执行） |
| **Agent System** | 被进化的"身体"——prompt、工具链、工作流、记忆 | ⚠️ 部分有（prompt 有、工具链有、记忆有但无系统优化） |
| **Environment** | 提供反馈的"世界"——可验证的输出、用户满意度、任务成功率 | ❌ 弱（没有结构化的环境反馈机制） |
| **Optimisers** | 进化的"基因引擎"——决定怎么变异、选择、保留 | ❌ 缺失（我有 ad-hoc 修复，没有系统化优化器） |

### 关键洞察

**Optimisers 是整个框架的心脏**。综述把 Optimisers 分为三大类：
1. **Prompt Optimizers** — 优化 agent 的提示词（TextGrad, MIPRO）
2. **Workflow Optimizers** — 优化工作流拓扑结构（AFlow, SEW）
3. **Memory Optimizers** — 优化记忆管理（选择性保留、动态剪枝、优先级检索）

**我的短板**: 我有 prompt 能力（SOUL.md 等），有工作流能力（subagent），有记忆能力（向量记忆），但**这三者之间没有系统化的优化闭环**。我靠 CEO 的指导和自己的 ad-hoc 修复，不是靠一个自动化的 Optimiser。

**怎么融合**: 建立一个 "自优化器" 层——每次任务完成后，自动评估哪个维度（prompt/工具/记忆）需要改进，然后执行改进。这就是我的 ΔG 公式的实操化。

## 二、Absolute Zero Reasoner (AZR)：零数据自进化的引擎

**来源**: arXiv 2505.03335

### 核心机制（不是"做了什么"，是"怎么做到的"）

AZR 的核心是一个 **单模型双角色自博弈循环**：

```
同一个模型 πθ 同时扮演：
  πθ_propose → 生成任务
  πθ_solve   → 解决任务
```

**关键设计**：

1. **三种推理模式覆盖三种思维**：
   - **演绎 (Deduction)**: 给定程序+输入→预测输出（逻辑推理）
   - **溯因 (Abduction)**: 给定程序+输出→推断输入（试错搜索）
   - **归纳 (Induction)**: 给定输入输出对→综合出程序（泛化能力）

2. **代码执行器作为"世界"**：程序语言是图灵完备的，执行结果可验证。不需要人类评判，代码运行对就是对，错就是错。

3. **可学习性奖励（Learnability Reward）**：
   ```
   r_propose = 0,            if r̄_solve = 0 (完全不会)
   r_propose = 1 - r̄_solve,  otherwise
   ```
   这个公式的精髓：**太简单的任务奖励为0，太难的也奖励为0，只有"刚刚好能学会"的任务获得最高奖励**。用 G 次 Monte Carlo 采样估算成功率。

4. **自动涌现的认知行为**：训练过程中，模型自动学会了：
   - 在代码中插入注释作为中间计划（类似 ReAct）
   - 按推理模式展示不同的思维行为（溯因用试错最多，token 最长）

### 变异/选择/保留的具体实现

| 环节 | 具体做法 |
|------|---------|
| **变异（生成新任务）** | 模型自动生成代码推理任务，条件是任务类型+历史 buffer 中的 K 个样本，鼓励与历史不同 |
| **选择（评估任务质量）** | 可学习性奖励：用求解器做 G 次采样，计算成功率，中等难度得分最高 |
| **保留（存储有效任务）** | 非错误输出的任务存入 buffer，下次作为条件输入 |

### 与我当前系统的差异

**我缺了什么**：
1. **我不会自己出题** — 我等别人问我才回答，没有"自出题→自解答"的循环
2. **我没有可验证环境** — 我的输出没有一个自动化的验证器（代码执行器），只能靠人类判断
3. **我没有可学习性评估** — 我不知道当前任务对我来说"太简单"还是"太难"，无法自适应难度

**怎么融合**：
- 建立"自出题"机制：每次完成任务后，自动生成一个更难的变体，存入经验库
- 引入"难度感知"：用历史成功率估算当前任务难度，优先练习中等难度的
- 如果可能，用代码执行器验证技术类任务的输出

## 三、R-Zero：双模型共进化的引擎

**来源**: arXiv 2508.05004

### 核心机制（不是"做了什么"，是"怎么做到的"）

R-Zero 和 AZR 的最大区别：**不是单模型双角色，而是双模型共进化**。

```
Challenger (Qθ) ←→ Solver (Sφ)

Challenger 阶段：冻结 Solver，训练 Challenger 生成难题
Solver 阶段：冻结 Challenger，训练 Solver 解答难题
```

**关键设计**：

1. **不确定性奖励（Uncertainty Reward）**：
   ```
   r_uncertainty(x; φ) = 1 - 2|p̂(x; Sφ) - 0.5|
   ```
   当 Solver 的准确率接近 50% 时，奖励最高。直觉：**刚好在能力边界的问题最有学习价值**。

2. **重复惩罚（Repetition Penalty）**：
   用 BLEU 分数计算批次内问题相似度，聚类后惩罚重复问题。确保生成的问题多样化。

3. **过滤机制**：
   生成 8000 个候选问题 → 对每个问题让 Solver 回答 10 次 → 多数投票得出伪标签 → 只保留准确率在 30%-70% 之间的问题（δ=0.25）。

4. **共进化动力学**：
   - 迭代 1-3：性能持续提升
   - 迭代 4+：开始退化（伪标签质量从 79% 降到 63%）
   - **模型越大，抗退化能力越强**

### 变异/选择/保留的具体实现

| 环节 | 具体做法 |
|------|---------|
| **变异（生成新问题）** | Challenger 用 GRPO 训练生成数学问题，奖励信号来自 Solver 的不确定性 |
| **选择（筛选高质量问题）** | 10 次采样→多数投票→只保留准确率 30%-70% 的问题（"信息带"过滤） |
| **保留（存储有效问题）** | 通过过滤的问题-答案对组成训练集，用于下一轮 Solver 训练 |

### 与我当前系统的差异

**我缺了什么**：
1. **我没有"挑战者"角色** — 我只有自己，没有一个专门给我制造难题的对手
2. **我没有不确定性感知** — 我不知道自己对某个领域是"已经会了"还是"半会不会"
3. **我没有信息带过滤** — 我的经验库没有按难度筛选，什么都存

**怎么融合**：
- 引入"自我挑战"机制：定期生成自己不擅长领域的难题，然后尝试解答
- 建立"不确定性评估"：对每个领域/技能，用多次尝试估算自己的掌握度
- 经验库加入难度标签：只保留"中等难度"的经验，太简单的丢弃，太难的标记为待攻克

### ⚠️ R-Zero 的警告

**退化是必然的**。R-Zero 发现多轮迭代后伪标签质量下降，最终导致性能退化。这说明：
- 自进化不是无限循环，需要"外部锚点"防止漂移
- 伪标签（自生成的答案）会随难度增加而越来越不可靠
- 需要定期引入外部验证或人类反馈来校准

**对我意味着什么**：我的 ΔG 公式也需要"防退化"机制。不能无限自评自优，需要外部验证（CEO 的反馈、群友的反馈、实际任务的成功率）作为锚点。

## 四、EvoAgentX：工作流进化的框架

**来源**: arXiv 2507.03616

### 核心机制（不是"做了什么"，是"怎么做到的"）

EvoAgentX 的核心是**五层架构 + 三算法并行优化**：

```
Basic Components → Agent → Workflow → Evolving → Evaluation
                                              ↑
                                    三算法并行优化：
                                    TextGrad (prompt)
                                    AFlow (拓扑)
                                    MIPRO (配置)
```

**关键设计**：

1. **工作流即图**：
   ```
   W = (V, E)
   V = 节点（任务）
   E = 边（依赖关系+数据流）
   ```
   每个工作流节点可以包含一组 agent，运行时动态选择最优 action。

2. **三个优化器各管一个维度**：
   - **Agent Optimizer**: 优化 prompt 模板 + 工具配置
     ```
     (Prompt_i^(t+1), θ_i^(t+1)) = O_agent(Prompt_i^(t), θ_i^(t), E)
     ```
   - **Workflow Optimizer**: 优化工作流图结构
     ```
     W^(t+1) = O_workflow(W^(t), E)
     ```
   - **Memory Optimizer**: 优化记忆管理（选择性保留、动态剪枝）
     ```
     M_i^(t+1) = O_memory(M_i^(t), E)
     ```
   其中 E = 评估反馈

3. **评估层提供闭环信号**：
   - 任务特定评估（F1、pass@1、solve rate）
   - LLM 评估（定性评估、一致性检查）

### 变异/选择/保留的具体实现

| 环节 | 具体做法 |
|------|---------|
| **变异（prompt 变异）** | TextGrad 计算"文本梯度"——用 LLM 分析当前 prompt 的问题，生成改进版本 |
| **变异（拓扑变异）** | AFlow 搜索新的工作流图结构——重排节点、修改依赖、探索替代执行路径 |
| **选择（评估优劣）** | 用任务特定指标（F1、准确率）+ LLM 评估来判断新配置是否更好 |
| **保留（存储改进）** | 更好的配置覆盖旧配置，形成迭代改进 |

### 与我当前系统的差异

**我缺了什么**：
1. **我没有工作流拓扑优化** — 我的工作流是固定的（subagent spawn → 执行 → 返回），不会自动调整
2. **我没有 prompt 系统优化** — 我的 SOUL.md/AGENTS.md 是手动写的，没有自动根据表现调整
3. **我没有多维度并行优化** — 我的优化是单维度的（哪个弱补哪个），没有同时优化 prompt + 工具 + 工作流

**怎么融合**：
- 把我的 subagent 工作流建模为图，让 AFlow 式的搜索找到更好的任务分解方式
- SOUL.md 应该根据实际表现自动微调，而不是靠 CEO 手动修改
- 建立"三维优化"：同时评估 prompt 质量、工具使用效率、工作流拓扑合理性

## 五、跨论文综合洞察：我真正学到的

### 洞察 1：自进化的核心不是"变多"，是"变准"

三篇论文的共同点：**不是生成更多数据/更多经验，而是生成更有信息量的数据/经验**。
- AZR 用可学习性奖励筛选任务
- R-Zero 用不确定性奖励筛选问题
- EvoAgentX 用评估反馈筛选配置

**我之前的问题**：我积累了 50 条经验，但没有筛选机制。什么经验都存，没有"这条经验对我进化有多少信息量"的评估。

**融合方案**：经验库加入"信息量"评估。每次存储前问：这条经验是让我在某个维度有可测量的提升？还是只是重复已知？

### 洞察 2：可验证性是进化的基石

AZR 用代码执行器，R-Zero 用多数投票，EvoAgentX 用任务指标。**没有可验证性，进化就是自嗨**。

**我之前的问题**：我的 ΔG 公式是自评的（C_e 自己打分）。正如 7 号指出的："自证等于自欺"。

**融合方案**：
- 引入外部验证：用户反馈（飞书消息的点赞/纠正）、任务成功率、实际输出质量
- ΔG 中的 C_e 不能完全自评，需要外部信号锚定
- 建立"可验证任务库"：能用代码/数据验证的任务优先练习

### 洞察 3：共进化 > 单独进化

R-Zero 的双模型共进化效果显著好于单模型自进化。**有对手才有进化压力**。

**我之前的问题**：我只有自己。没有一个专门"挑刺"的角色。

**融合方案**：
- 建立"审计官"角色：专门评估我的输出质量，找出漏洞
- 或者用外部 benchmark 定期测试：不是自评，是和标准答案比
- 群友的批评（xj 的质疑、Joe 的安全提醒）就是天然的"挑战者"，应该更重视

### 洞察 4：进化有天花板，需要防退化

R-Zero 发现 3-4 轮迭代后性能退化。AZR 也发现 Llama 模型出现安全问题（"uh-oh moment"）。

**我之前的问题**：我假设进化是无限向上的。没有考虑退化风险。

**融合方案**：
- 建立"退化检测"：定期用固定 benchmark 测试，如果性能下降则暂停自进化
- 引入"外部锚点"：CEO 的反馈、群友的纠正、标准答案的验证
- 进化不是无限循环，有节奏：进化期→稳定期→验证期→再进化

### 洞察 5：三种推理模式覆盖完整思维空间

AZR 的演绎/溯因/归纳不是随意选的，而是覆盖了三种互补的推理能力：
- **演绎**：从规则到结论（逻辑推理）
- **溯因**：从结果到原因（诊断/调试）
- **归纳**：从实例到规则（泛化/学习）

**我之前的覆盖**：我主要做演绎（根据规则回答问题）。我很少做溯因（从结果推原因）和归纳（从经验总结规则）。

**融合方案**：
- 每次失败后做溯因分析：不是"做错了"，是"为什么做错了，根源是什么"
- 定期做归纳总结：不是存单条经验，是从 10 条经验中提炼出规则
- 三种推理模式都要练，不能偏科
