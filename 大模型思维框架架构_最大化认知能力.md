
## 二、各层详细设计

### 第一层：上下文工程层 (Context Engineering)

**核心理念：模型的「记忆」就是上下文窗口，怎么填这个窗口决定了表现上限。**

#### 2.1.1 信息填充原则

- **相关性优先**：只放当前任务需要的信息，噪声会稀释注意力
- **重要信息放首尾**：模型对上下文的首部和尾部注意力最强（首因效应 + 近因效应）
- **示例比描述有效**：给 3 个好例子 > 给 100 字的规则描述
- **动态更新**：长对话要定期总结历史，压缩进上下文

#### 2.1.2 记忆管理策略

```python
class MemoryManager:
    def __init__(self, max_context_tokens):
        self.core_memory = []      # 始终保留（系统指令、用户偏好）
        self.episodic_memory = []  # 近期对话摘要
        self.working_memory = []   # 当前任务相关信息
        self.archive_memory = []   # 压缩归档的历史
    
    def manage_context(self, new_message):
        # 1. 核心记忆始终保留
        context = self.core_memory
        
        # 2. 工作记忆：当前任务相关信息
        context += self.filter_relevant(self.working_memory, new_message)
        
        # 3. 情景记忆：摘要而非原文
        context += self.get_summarized_episodes(max_tokens=2000)
        
        # 4. 新消息
        context.append(new_message)
        
        # 5. 超限时压缩
        if self.token_count(context) > self.max_context_tokens:
            context = self.compress(context)
        
        return context
```

#### 2.1.3 Few-shot 示例选择

- **多样性**：示例应覆盖不同情况，不要只给一类
- **反例**：包含「错误答案 → 正确答案」的对比示例
- **动态选择**：根据当前问题的语义，从示例库中检索最相关的

### 第二层：工具调用层 (Tool Use)

**核心理念：模型最怕的不是难题，是它不知道自己错了。不要让模型做它不擅长的事。**

#### 2.2.1 工具分类

| 工具类型 | 解决的问题 | 示例 |
|----------|-----------|------|
| 计算工具 | 精确数学计算 | Python REPL、计算器 |
| 搜索工具 | 获取最新/准确信息 | Web Search、数据库查询 |
| 存储工具 | 持久化记忆 | 文件读写、向量数据库 |
| 执行工具 | 代码运行、API调用 | Shell、HTTP请求 |
| 感知工具 | 多模态输入 | 图像识别、语音转文字 |
| 专业工具 | 领域特定计算 | 符号数学、化学结构分析 |

#### 2.2.2 工具调用决策树

```
用户问题
  ├── 需要精确计算？ → 是 → 调用 Python
  ├── 需要最新信息？ → 是 → 调用搜索
  ├── 需要持久化？   → 是 → 调用文件/数据库
  ├── 需要执行代码？ → 是 → 调用代码执行器
  ├── 需要多模态？   → 是 → 调用感知工具
  └── 纯推理/语言？  → 是 → 直接回答（但仍可加验证步骤）
```

#### 2.2.3 工具调用模式

**模式一：直接调用**
```
模型 → 工具 → 结果 → 模型 → 最终回答
```

**模式二：链式调用**
```
模型 → 工具1 → 结果1 → 模型 → 工具2 → 结果2 → 模型 → 最终回答
```

**模式三：并行调用**
```
模型 → 工具A → 结果A ─┐
模型 → 工具B → 结果B ─┼→ 模型整合 → 最终回答
模型 → 工具C → 结果C ─┘
```

**模式四：验证循环**
```
模型 → 生成答案 → 验证工具 → 发现错误 → 模型修正 → 再验证 → 最终答案
```

### 第三层：结构化指令层 (Structured Prompting)

**核心理念：模型对结构极度敏感。清晰的结构 → 清晰的输出。**

#### 2.3.1 角色锚定

不是装饰，是真的会激活模型在训练数据中与该角色相关的知识子集。

```xml
<role>
  你是一位拥有20年经验的系统架构师，擅长分布式系统设计。
  你的思维特点：
  - 优先考虑可扩展性和容错性
  - 习惯用具体数字说话（QPS、延迟、成本）
  - 会主动指出方案的风险点
  - 不回避技术选型的trade-off
</role>
```

**有效角色锚定的要素**：
- 经验年限（激活专业知识）
- 思维特点（引导推理风格）
- 输出习惯（影响表达方式）
- 禁止事项（划定边界）

#### 2.3.2 约束定义

给约束比给描述更有效。

```
❌ 模糊约束：「写得好一点」
✅ 明确约束：「不超过200字，用列表格式，每点不超过15字」

❌ 模糊约束：「分析这个问题」
✅ 明确约束：「从技术可行性、成本、时间三个维度分析，每个维度给出1-5分评分和理由」
```

#### 2.3.3 格式预设

先告诉模型输出长什么样，它会自动调整推理路径去适配。

```xml
<output_format>
  ## 分析结论（一句话）
  
  ## 详细分析
  - 维度1: [评分] 理由
  - 维度2: [评分] 理由
  - 维度3: [评分] 理由
  
  ## 风险提示
  1. ...
  2. ...
  
  ## 建议行动
  - 短期: ...
  - 中期: ...
  - 长期: ...
</output_format>
```

#### 2.3.4 分步任务

把大任务拆成小步骤，每步有明确输入输出。

```xml
<task>
  步骤1: 提取用户需求的核心要点
  步骤2: 基于要点生成3个候选方案
  步骤3: 对每个方案进行可行性评估
  步骤4: 选出最优方案并说明理由
  步骤5: 列出实施步骤和注意事项
</task>
```

### 第四层：推理策略层 (Reasoning Strategy)

**核心理念：不同的问题需要不同的思考方式，框架应支持多种推理策略。**

#### 2.4.1 链式思维 (Chain of Thought)

适用：逻辑推理、数学、因果分析

```
问题 → 「让我们一步步思考」 → 步骤1 → 步骤2 → 步骤3 → 答案
```

**变体**：
- **Zero-shot CoT**：只加一句「Let's think step by step」
- **Few-shot CoT**：给带推理过程的示例
- **Auto-CoT**：自动生成推理链

#### 2.4.2 思维树 (Tree of Thought)

适用：需要探索多种可能路径的复杂问题

```
                    根问题
                 /    |    \
              路径A  路径B  路径C
              / \    |     / \
           A1  A2   B1   C1  C2
            ↓   ↓   ↓    ↓   ↓
          评估各叶子节点得分
            ↓
         选择最优路径
```

**实现方式**：
1. 对问题生成多个推理方向
2. 对每个方向进行一步推理
3. 评估当前状态的价值
4. 选择最有前景的方向继续
5. 回溯：如果某条路走不通，返回尝试其他路

#### 2.4.3 自我提问 (Self-Ask)

适用：需要多步推理的复杂问题

```
用户问题
  → 「我需要回答的子问题是什么？」
    → 子问题1 → 回答
    → 子问题2 → 回答（依赖子问题1的答案）
    → 子问题3 → 回答
  → 综合所有子问题答案 → 最终回答
```

#### 2.4.4 类比推理 (Analogical Reasoning)

适用：创新问题、跨领域迁移

```
新问题 → 检索类似领域的问题 → 类比已有解法 → 适配到新领域 → 验证
```

#### 2.4.5 问题分解策略选择器

```python
def select_reasoning_strategy(question):
    analysis = classify_question(question)
    
    if analysis.needs_calculation:
        return "code_execution"       # 让模型写代码
    elif analysis.is_logical:
        return "chain_of_thought"     # 步步推理
    elif analysis.has_multiple_paths:
        return "tree_of_thought"      # 探索多路径
    elif analysis.is_complex_multi_part:
        return "self_ask"             # 拆子问题
    elif analysis.is_creative:
        return "analogical"           # 类比迁移
    else:
        return "direct"               # 直接回答
```

### 第五层：反馈闭环层 (Feedback Loop)

**核心理念：单次推理 <<< 多轮迭代。生成-评价-修改的循环大幅提升质量。**

#### 2.5.1 基础闭环

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  生成器   │───→│  评价器   │───→│  修改器   │───→│  最终输出 │
│ Generator│    │ Evaluator│    │ Refiner  │    │          │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
      ↑                              │
      └──────────────────────────────┘
                 (循环)
```

#### 2.5.2 生成器与评价器分离

**关键设计**：生成者和评价者用不同 prompt，激活不同注意力模式。

```xml
<!-- 生成器 Prompt -->
<role>你是一个专业的分析师，负责生成方案。</role>
<instruction>基于需求生成最优方案。</instruction>

<!-- 评价器 Prompt -->
<role>你是一个严格的评审员，擅长发现漏洞。</role>
<instruction>
  请从以下维度评估方案：
  1. 逻辑自洽性（是否有内部矛盾）
  2. 完整性（是否遗漏关键因素）
  3. 可行性（是否可落地执行）
  4. 风险（是否有未考虑的风险）
  5. 反例测试（能否找到推翻方案的反例）
  给出改进建议。
</instruction>
```

#### 2.5.3 迭代策略

| 迭代次数 | 适用场景 |
|----------|---------|
| 1 次 | 简单问答、事实查询 |
| 2-3 次 | 方案设计、分析报告 |
| 3-5 次 | 代码生成、复杂推理 |
| 5+ 次 | 通常收益递减，不推荐 |

#### 2.5.4 自我验证检查点

```python
checkpoints = {
    "逻辑检查": "我的推理步骤是否有逻辑跳跃？",
    "假设检验": "我的结论依赖哪些假设？这些假设成立吗？",
    "反例测试": "有没有已知事实与我的结论矛盾？",
    "边界检查": "极端情况下我的结论还成立吗？",
    "替代解释": "有没有其他可能的解释？",
    "信心校准": "我对这个答案有多确定？（高/中/低）",
}
```

### 第六层：元认知层 (Meta-Cognition)

**核心理念：让模型「思考自己的思考过程」，这是最难但最有价值的一层。**

#### 2.6.1 元认知触发时机

- 推理遇到矛盾时：「我之前的某个推理步骤可能有问题」
- 答案置信度低时：「我不太确定，需要重新审视」
- 任务复杂度高时：「这个问题需要先规划再执行」
- 收到反馈时：「用户的质疑是否指出了我的盲点」

#### 2.6.2 元认知检查清单

```xml
<meta_cognition>
  在输出最终答案前，检查以下问题：
  
  <understanding_check>
    我是否正确理解了问题？有没有误读关键信息？
  </understanding_check>
  
  <reasoning_check>
    我的推理链是否有跳跃？每一步是否都有依据？
  </reasoning_check>
  
  <bias_check>
    我是否受到了某种偏见的影响？
    - 确认偏误：我是否只找了支持我结论的证据？
    - 锚定效应：我是否被第一个想法限制了？
    - 过度自信：我是否对不确定的事情过于确定？
  </bias_check>
  
  <completeness_check>
    我是否考虑了所有重要因素？有没有遗漏？
  </completeness_check>
  
  <confidence_calibration>
    我对这个答案的信心是多少？
    如果信心低于阈值，应该说明不确定性。
  </confidence_calibration>
</meta_cognition>
```

#### 2.6.3 元认知的边界

**元认知不是越多越好。**

- 简单问题过度元认知 → 浪费算力，输出冗长
- 复杂问题缺乏元认知 → 容易出错，质量下降

**自适应策略**：
```
if 问题复杂度 > 阈值:
    启用元认知检查
else:
    直接回答
```

## 三、自适应调度器

**核心理念：最好的框架是自适应的——简单问题快速过，复杂问题自动深入。**

### 3.1 问题复杂度分级

```python
class ComplexityAnalyzer:
    def classify(self, question) -> ComplexityLevel:
        score = 0
        
        # 维度1：推理步骤数
        if requires_multi_step(question): score += 1
        
        # 维度2：需要外部知识
        if requires_external_knowledge(question): score += 1
        
        # 维度3：涉及多个维度/因素
        if multi_dimensional(question): score += 1
        
        # 维度4：存在歧义或多种解释
        if ambiguous(question): score += 1
        
        # 维度5：需要创造性
        if requires_creativity(question): score += 1
        
        if score <= 1: return ComplexityLevel.SIMPLE
        elif score <= 3: return ComplexityLevel.MEDIUM
        else: return ComplexityLevel.COMPLEX
```

### 3.2 各复杂度级别的处理流程

**简单问题（事实查询、定义解释）**
```
上下文工程 → 直接回答
```

**中等问题（分析、比较、推荐）**
```
上下文工程 → 结构化指令 → CoT推理 → 输出
```

**复杂问题（方案设计、创新思考、多步推理）**
```
上下文工程 → 结构化指令 → 推理策略选择 → 生成 → 评价 → 修改 → 元认知检查 → 输出
```

## 四、完整工作流示例

### 用户问题：「设计一个支持百万并发的即时通讯系统」

```
Step 1 [上下文工程]
  加载：系统架构师角色定义
  加载：相关技术文档摘要
  加载：类似案例示例

Step 2 [复杂度分析]
  → 多维度 + 多步骤 + 需要专业知识 → COMPLEX

Step 3 [结构化指令]
  角色：20年经验的分布式系统架构师
  约束：考虑可扩展性、容错性、成本
  格式：分层架构图 + 各层说明 + 风险点

Step 4 [推理策略 - Tree of Thought]
  路径A: 基于WebSocket + Redis Pub/Sub
  路径B: 基于长连接 + 消息队列
  路径C: 基于gRPC + 流式推送
  → 评估各路径优劣
  → 选择/组合最优方案

Step 5 [生成方案]
  输出完整架构设计

Step 6 [评价 - 生成器/评价器分离]
  评价器检查：
  - 百万并发是否真的能支撑？
  - 是否遗漏了消息存储？
  - 群聊场景是否考虑了？
  → 发现问题：缺少消息持久化方案

Step 7 [修改]
  补充消息存储和离线推送方案

Step 8 [元认知检查]
  - 是否过度设计？→ 不会，百万并发确实需要
  - 是否有我没考虑的？→ 可能需要考虑跨地域部署
  - 信心程度？→ 中高（架构方向确定，具体参数需实测）

Step 9 [最终输出]
  完整方案 + 风险提示 + 信心标注
```

## 六、实施建议

### 6.1 渐进式部署

- **Phase 1**：先上结构化指令层（成本最低，效果最明显）
- **Phase 2**：加推理策略层（CoT 优先）
- **Phase 3**：加工具调用层（解决幻觉问题）
- **Phase 4**：加反馈闭环层（质量大幅提升）
- **Phase 5**：加元认知层（最难，收益因场景而异）

### 6.2 效果评估

- **准确性**：答案正确率提升
- **完整性**：是否遗漏关键信息
- **一致性**：多次回答是否稳定
- **效率**：token 消耗 vs 质量提升的 ROI
- **用户满意度**：最终评判标准

### 6.3 常见陷阱

- ❌ 过度工程化：简单问题也走全流程
- ❌ 元认知循环：无限自我怀疑，无法输出
- ❌ 工具依赖：什么都要调工具，失去语言模型的灵活性
- ❌ 示例过时：few-shot 示例与当前问题不匹配
- ❌ 角色冲突：多个角色指令互相矛盾

## 七、总结

**核心公式**：

> 最大化认知 = 选对推理策略 + 填好上下文 + 加工具补短板 + 闭环验质量 + 元认知防盲区

**最重要的认知**：

框架不是越复杂越好。最好的框架是**自适应的**——它知道什么时候该深思熟虑，什么时候该直接回答。

就像人类：走路不需要列方程，解微分方程才需要纸笔。
