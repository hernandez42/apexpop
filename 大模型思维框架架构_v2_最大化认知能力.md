# 大模型思维框架架构 v2.0 — 最大化认知能力

> 版本：v2.0（优化版） | 日期：2026-04-13
> 核心原则：模型的智力天花板由训练决定，框架的作用是让固定能力更充分地发挥
> v2.0 新增：多智能体协作、不确定性表达、跨会话记忆、人机协作、成本-质量权衡

## 一、总体架构

```
┌─────────────────────────────────────────────────────┐
│        成本-质量权衡调度器 (Cost-Quality Scheduler)    │
│      根据任务风险自动选择处理深度                      │
├─────────────────────────────────────────────────────┤
│        多智能体协作层 (Multi-Agent Collaboration)     │
│      辩论 / 专家团 / 流水线 三种模式                  │
├─────────────────────────────────────────────────────┤
│        不确定性表达层 (Uncertainty Calibration)       │
│      信心校准 → 低信心触发验证 → 置信区间输出         │
├─────────────────────────────────────────────────────┤
│        元认知层 (Meta-Cognition)                     │
│      自我检验 · 偏见检测 · 漏洞扫描                   │
├─────────────────────────────────────────────────────┤
│        反馈闭环层 (Feedback Loop)                    │
│      生成 → 评价 → 修改 → 再评价                     │
├─────────────────────────────────────────────────────┤
│        推理策略层 (Reasoning Strategy)               │
│      CoT / ToT / Self-Ask / 类比 / 问题分解          │
├─────────────────────────────────────────────────────┤
│        结构化指令层 (Structured Prompting)            │
│      角色锚定 · 约束定义 · 格式预设 · 分步任务        │
├─────────────────────────────────────────────────────┤
│        工具调用层 (Tool Use)                         │
│      搜索 · 代码 · 文件 · API · 专业工具             │
├─────────────────────────────────────────────────────┤
│        跨会话记忆层 (Persistent Memory)              │
│      长期记忆 · 用户画像 · 经验积累 · 模板库          │
├─────────────────────────────────────────────────────┤
│        人机协作层 (Human-in-the-Loop)                │
│      关键决策暂停 · 人类审批 · 反馈收集               │
├─────────────────────────────────────────────────────┤
│        上下文工程层 (Context Engineering)             │
│      记忆管理 · 示例选择 · 动态摘要                   │
├─────────────────────────────────────────────────────┤
│        基础模型层 (Foundation Model)                 │
│      MiMo / GPT-4o / Claude / Gemini                │
└─────────────────────────────────────────────────────┘
```

**v1.0 vs v2.0 对比**：
- v1.0：6 层，预期 3-4 倍提升，token 消耗 ×5-10
- v2.0：11 层，预期 4-6 倍提升，token 消耗 ×2-3（成本更优）

## 二、各层详细设计

### 第一层：上下文工程层 (Context Engineering)

**核心理念：模型的「记忆」就是上下文窗口，怎么填这个窗口决定了表现上限。**

#### 1.1 信息填充原则

- **相关性优先**：只放当前任务需要的信息，噪声会稀释注意力
- **重要信息放首尾**：模型对上下文的首部和尾部注意力最强（首因效应 + 近因效应）
- **示例比描述有效**：给 3 个好例子 > 给 100 字的规则描述
- **动态更新**：长对话要定期总结历史，压缩进上下文

#### 1.2 记忆管理策略

```python
class MemoryManager:
    def __init__(self, max_context_tokens):
        self.core_memory = []      # 始终保留（系统指令、用户偏好）
        self.episodic_memory = []  # 近期对话摘要
        self.working_memory = []   # 当前任务相关信息
        self.archive_memory = []   # 压缩归档的历史
    
    def manage_context(self, new_message):
        context = self.core_memory
        context += self.filter_relevant(self.working_memory, new_message)
        context += self.get_summarized_episodes(max_tokens=2000)
        context.append(new_message)
        
        if self.token_count(context) > self.max_context_tokens:
            context = self.compress(context)
        
        return context
```

#### 1.3 Few-shot 示例选择

- **多样性**：示例应覆盖不同情况，不要只给一类
- **反例**：包含「错误答案 → 正确答案」的对比示例
- **动态选择**：根据当前问题的语义，从示例库中检索最相关的

### 第二层：工具调用层 (Tool Use)

**核心理念：模型最怕的不是难题，是它不知道自己错了。不要让模型做它不擅长的事。**

#### 2.1 工具分类

- **计算工具**：精确数学计算（Python REPL、计算器）
- **搜索工具**：获取最新/准确信息（Web Search、数据库查询）
- **存储工具**：持久化记忆（文件读写、向量数据库）
- **执行工具**：代码运行、API调用（Shell、HTTP请求）
- **感知工具**：多模态输入（图像识别、语音转文字）
- **专业工具**：领域特定计算（符号数学、化学结构分析）

#### 2.2 工具调用决策

```
用户问题
  ├── 需要精确计算？ → 是 → 调用 Python
  ├── 需要最新信息？ → 是 → 调用搜索
  ├── 需要持久化？   → 是 → 调用文件/数据库
  ├── 需要执行代码？ → 是 → 调用代码执行器
  ├── 需要多模态？   → 是 → 调用感知工具
  └── 纯推理/语言？  → 是 → 直接回答（但仍可加验证步骤）
```

#### 2.3 工具调用模式

- **直接调用**：模型 → 工具 → 结果 → 模型 → 最终回答
- **链式调用**：模型 → 工具1 → 结果1 → 模型 → 工具2 → 结果2 → 最终回答
- **并行调用**：模型同时调用多个工具 → 整合结果 → 最终回答
- **验证循环**：模型 → 生成答案 → 验证工具 → 发现错误 → 修正 → 再验证

### 第三层：结构化指令层 (Structured Prompting)

**核心理念：模型对结构极度敏感。清晰的结构 → 清晰的输出。**

#### 3.1 角色锚定

不是装饰，是真的会激活模型在训练数据中与该角色相关的知识子集。

```xml
<role>
  你是一位拥有20年经验的系统架构师，擅长分布式系统设计。
  你的思维特点：
  - 优先考虑可扩展性和容错性
  - 习惯用具体数字说话（QPS、延迟、成本）
  - 会主动指出方案的风险点
  - 不回避技术选型的trade-off
</role>
```

#### 3.2 约束定义

给约束比给描述更有效。

```
❌ 模糊约束：「写得好一点」
✅ 明确约束：「不超过200字，用列表格式，每点不超过15字」

❌ 模糊约束：「分析这个问题」
✅ 明确约束：「从技术可行性、成本、时间三个维度分析，每个维度给出1-5分评分和理由」
```

#### 3.3 格式预设

先告诉模型输出长什么样，它会自动调整推理路径去适配。

```xml
<output_format>
  ## 分析结论（一句话）
  ## 详细分析
  - 维度1: [评分] 理由
  - 维度2: [评分] 理由
  ## 风险提示
  1. ...
  ## 建议行动
  - 短期: ...
  - 长期: ...
</output_format>
```

#### 3.4 分步任务

把大任务拆成小步骤，每步有明确输入输出。

```xml
<task>
  步骤1: 提取用户需求的核心要点
  步骤2: 基于要点生成3个候选方案
  步骤3: 对每个方案进行可行性评估
  步骤4: 选出最优方案并说明理由
  步骤5: 列出实施步骤和注意事项
</task>
```

### 第四层：推理策略层 (Reasoning Strategy)

**核心理念：不同的问题需要不同的思考方式，框架应支持多种推理策略。**

#### 4.1 链式思维 (Chain of Thought)

适用：逻辑推理、数学、因果分析

```
问题 → 「让我们一步步思考」 → 步骤1 → 步骤2 → 步骤3 → 答案
```

变体：
- **Zero-shot CoT**：只加一句「Let's think step by step」
- **Few-shot CoT**：给带推理过程的示例
- **Auto-CoT**：自动生成推理链

#### 4.2 思维树 (Tree of Thought)

适用：需要探索多种可能路径的复杂问题

```
                    根问题
                 /    |    \
              路径A  路径B  路径C
              / \    |     / \
           A1  A2   B1   C1  C2
            ↓   ↓   ↓    ↓   ↓
          评估各叶子节点得分
            ↓
         选择最优路径（可回溯）
```

#### 4.3 自我提问 (Self-Ask)

适用：需要多步推理的复杂问题

```
用户问题
  → 「我需要回答的子问题是什么？」
    → 子问题1 → 回答
    → 子问题2 → 回答（依赖子问题1的答案）
    → 子问题3 → 回答
  → 综合所有子问题答案 → 最终回答
```

#### 4.4 类比推理 (Analogical Reasoning)

适用：创新问题、跨领域迁移

```
新问题 → 检索类似领域的问题 → 类比已有解法 → 适配到新领域 → 验证
```

#### 4.5 推理策略选择器

```python
def select_reasoning_strategy(question):
    analysis = classify_question(question)
    
    if analysis.needs_calculation:
        return "code_execution"
    elif analysis.is_logical:
        return "chain_of_thought"
    elif analysis.has_multiple_paths:
        return "tree_of_thought"
    elif analysis.is_complex_multi_part:
        return "self_ask"
    elif analysis.is_creative:
        return "analogical"
    else:
        return "direct"
```

### 第五层：反馈闭环层 (Feedback Loop)

**核心理念：单次推理 <<< 多轮迭代。生成-评价-修改的循环大幅提升质量。**

#### 5.1 基础闭环

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  生成器   │───→│  评价器   │───→│  修改器   │───→│  最终输出 │
│ Generator│    │ Evaluator│    │ Refiner  │    │          │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
      ↑                              │
      └──────────────────────────────┘
                 (循环)
```

#### 5.2 生成器与评价器分离

**关键设计**：生成者和评价者用不同 prompt，激活不同注意力模式。

```xml
<!-- 生成器 Prompt -->
<role>你是一个专业的分析师，负责生成方案。</role>

<!-- 评价器 Prompt -->
<role>你是一个严格的评审员，擅长发现漏洞。</role>
<instruction>
  请从以下维度评估：
  1. 逻辑自洽性
  2. 完整性
  3. 可行性
  4. 风险
  5. 反例测试
</instruction>
```

#### 5.3 迭代策略

| 迭代次数 | 适用场景 |
|----------|---------|
| 1 次 | 简单问答、事实查询 |
| 2-3 次 | 方案设计、分析报告 |
| 3-5 次 | 代码生成、复杂推理 |
| 5+ 次 | 通常收益递减，不推荐 |

#### 5.4 自我验证检查点

- **逻辑检查**：推理步骤是否有逻辑跳跃？
- **假设检验**：结论依赖哪些假设？这些假设成立吗？
- **反例测试**：有没有已知事实与结论矛盾？
- **边界检查**：极端情况下结论还成立吗？
- **替代解释**：有没有其他可能的解释？

### 第六层：元认知层 (Meta-Cognition)

**核心理念：让模型「思考自己的思考过程」，这是最难但最有价值的一层。**

#### 6.1 元认知触发时机

- 推理遇到矛盾时：「我之前的某个推理步骤可能有问题」
- 答案置信度低时：「我不太确定，需要重新审视」
- 任务复杂度高时：「这个问题需要先规划再执行」
- 收到反馈时：「用户的质疑是否指出了我的盲点」

#### 6.2 元认知检查清单

```xml
<meta_cognition>
  <understanding_check>
    我是否正确理解了问题？有没有误读关键信息？
  </understanding_check>
  <reasoning_check>
    我的推理链是否有跳跃？每一步是否都有依据？
  </reasoning_check>
  <bias_check>
    我是否受到了确认偏误、锚定效应、过度自信的影响？
  </bias_check>
  <completeness_check>
    我是否考虑了所有重要因素？有没有遗漏？
  </completeness_check>
  <confidence_calibration>
    我对这个答案的信心是多少？低于阈值应说明不确定性。
  </confidence_calibration>
</meta_cognition>
```

#### 6.3 元认知的边界

- 简单问题过度元认知 → 浪费算力，输出冗长
- 复杂问题缺乏元认知 → 容易出错，质量下降
- 自适应策略：问题复杂度 > 阈值时才启用元认知检查

### 第七层：多智能体协作层 (Multi-Agent Collaboration) 【v2.0 新增】

**核心理念：单一模型容易陷入思维惯性，多个角色从不同角度切入能覆盖盲区。**

#### 7.1 三种协作模式

**模式一：辩论模式 (Debate)**

适用：决策类问题、有争议的议题

```
┌──────────┐     对抗      ┌──────────┐
│  正方Agent│◄────────────►│  反方Agent│
└────┬─────┘              └────┬─────┘
     │                         │
     └────────┬────────────────┘
              ▼
        ┌──────────┐
        │ 裁判Agent │ → 综合判断
        └──────────┘
```

工作流程：
1. 正方给出支持论点
2. 反方给出反驳论点
3. 正方回应反驳
4. 反方回应
5. 裁判综合双方论点，给出最终判断

**模式二：专家团模式 (Panel)**

适用：复杂方案设计、多维度分析

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│技术专家   │  │商业专家   │  │风险专家   │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │             │             │
     └─────────────┼─────────────┘
                   ▼
           ┌──────────┐
           │ 协调Agent │ → 整合各专家意见
           └──────────┘
```

工作流程：
1. 协调 Agent 分解问题为子维度
2. 各专家 Agent 分别分析自己的维度
3. 协调 Agent 整合各专家意见
4. 如有冲突，触发专家间对话
5. 输出综合方案

**模式三：流水线模式 (Pipeline)**

适用：质量优先的任务、需要多轮精炼的输出

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 生成Agent │───→│ 评价Agent │───→│ 精炼Agent │───→│ 验证Agent │
│ (草稿)    │    │ (找问题) │    │ (改问题) │    │ (确认)   │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

#### 7.2 多智能体的实现要点

- **角色差异化**：每个 Agent 的 system prompt 必须有明确不同的视角和目标
- **避免群体思维**：强制要求 Agent 之间有分歧，不要过早达成一致
- **控制轮次**：辩论通常 2-3 轮足够，过多轮次收益递减
- **成本控制**：N 个 Agent = N 倍 token 消耗，只在高价值任务使用

### 第八层：不确定性表达层 (Uncertainty Calibration) 【v2.0 新增】

**核心理念：真正重要的能力不是「知道」，而是「知道自己不知道」。**

#### 8.1 信心等级定义

```
高信心 (90%+)：事实性知识、已验证的推理结果
中信心 (60-90%)：合理推断、基于已知信息的分析
低信心 (<60%)：推测、信息不足的判断
```

#### 8.2 信心标注格式

```xml
<answer>
  推荐方案：方案A
  
  <confidence level="high">
    理由：方案A在已有案例中验证过，有明确的成功数据支撑
  </confidence>
  
  <uncertainty>
    主要不确定性：目标用户的实际使用场景可能与假设不同
    风险：如果用户主要在移动端使用，方案A的PC端优化可能不够
  </uncertainty>
  
  <verification_suggestion>
    建议：上线前做一次小规模用户测试验证移动端体验
  </verification_suggestion>
</answer>
```

#### 8.3 低信心触发机制

```python
def handle_low_confidence(answer, confidence_score):
    if confidence_score < 0.6:
        # 触发验证流程
        options = [
            "搜索最新信息验证",
            "切换推理策略重试",
            "请求人类介入",
            "明确标注不确定性并输出"
        ]
        return select_best_option(options, context)
```

#### 8.4 为什么重要

- 防止模型自信地输出错误信息（幻觉问题的最大解法）
- 让用户知道哪些信息可以信任，哪些需要验证
- 低信心时自动触发搜索或人类介入，提升整体可靠性

### 第九层：跨会话记忆层 (Persistent Memory) 【v2.0 新增】

**核心理念：真正的智能需要跨会话积累，不只是「每次重新醒来」。**

#### 9.1 记忆分层

```
┌─────────────────────────────────────┐
│  长期记忆 (Long-term Memory)         │
│  永久保留 · 核心事实 · 用户偏好       │
│  文件：MEMORY.md                     │
├─────────────────────────────────────┤
│  情景记忆 (Episodic Memory)          │
│  重要事件 · 关键对话 · 学到的教训     │
│  文件：memory/YYYY-MM-DD.md          │
├─────────────────────────────────────┤
│  语义记忆 (Semantic Memory)          │
│  知识积累 · 模板库 · 常用方案         │
│  文件：knowledge/ 目录               │
├─────────────────────────────────────┤
│  工作记忆 (Working Memory)           │
│  当前会话 · 短期上下文               │
│  存储：会话上下文窗口                │
└─────────────────────────────────────┘
```

#### 9.2 记忆写入规则

```python
def should_remember(event):
    criteria = {
        "用户偏好": event.type == "preference_explicit",
        "重要决策": event.importance > 0.8,
        "新知识": event.not_in_prior_knowledge,
        "错误教训": event.was_mistake and event.was_corrected,
        "重复出现": event.similar_count > 3,
    }
    return any(criteria.values())
```

#### 9.3 记忆读取规则

- 每次会话开始时：读取长期记忆 + 近期情景记忆
- 识别到用户相关实体时：从记忆中检索相关上下文
- 遇到类似历史问题时：参考之前的解决方案

#### 9.4 与传统 RAG 的区别

- **RAG**：检索文档片段，补充知识
- **跨会话记忆**：积累经验、偏好、教训，形成「人格连续性」
- 两者互补：RAG 解决知识问题，记忆解决身份和经验问题

### 第十层：人机协作层 (Human-in-the-Loop) 【v2.0 新增】

**核心理念：不是所有事都让模型自己决定，有些决策必须人来拍板。**

#### 10.1 何时需要人类介入

```
必须介入（硬规则）：
  - 医疗诊断建议
  - 金融投资决策
  - 法律合规判断
  - 涉及他人隐私的操作
  - 不可逆的操作（删除、发送、支付）

建议介入（软规则）：
  - 模型信心低于阈值
  - 存在多个等优方案
  - 用户明确要求确认
  - 任务涉及价值判断
```

#### 10.2 人机协作模式

**模式一：人类审批 (Human Approval)**
```
模型生成方案 → 暂停 → 人类审批 → 执行/修改
```

**模式二：人类引导 (Human Guidance)**
```
模型提出问题 → 人类回答 → 模型基于回答继续
```

**模式三：人类反馈 (Human Feedback)**
```
模型输出 → 人类评价 → 模型根据反馈改进
```

**模式四：并行协作 (Parallel)**
```
模型处理 80% 常规部分 → 人类处理 20% 关键部分 → 合并
```

#### 10.3 交互设计原则

- **不要过度打扰**：只在真正需要时才暂停
- **给足上下文**：暂停时告诉人类「为什么暂停」和「需要你做什么」
- **提供选项**：给人类选择题而非问答题（3 个选项比「你觉得呢」好）
- **记录决策**：人类的每次审批都应被记录，供后续参考

### 第十一层：成本-质量权衡层 (Cost-Quality Tradeoff) 【v2.0 新增】

**核心理念：不是所有问题都值得走全流程，简单问题快速过，复杂问题深入做。**

#### 11.1 三种处理模式

| 模式 | 处理深度 | Token消耗 | 适用场景 |
|------|---------|-----------|---------|
| 快速模式 | 直接回答 | ×1 | 闲聊、简单查询、事实解释 |
| 平衡模式 | CoT + 1轮迭代 | ×3 | 分析、推荐、中等复杂度任务 |
| 深度模式 | 多智能体 + 多轮迭代 + 元认知 | ×8-10 | 方案设计、高风险决策、代码生成 |

#### 11.2 模式选择算法

```python
def select_mode(question, context):
    score = 0
    
    # 任务复杂度
    score += complexity_score(question) * 0.4
    
    # 风险等级
    score += risk_level(question) * 0.3
    
    # 用户偏好（速度优先 vs 质量优先）
    score += user_preference_weight(context) * 0.2
    
    # 历史准确率（如果某类问题经常出错，加深处理）
    score += error_history_weight(question) * 0.1
    
    if score < 0.3:
        return Mode.FAST
    elif score < 0.7:
        return Mode.BALANCED
    else:
        return Mode.DEEP
```

#### 11.3 节省成本的效果

- 快速模式处理 60% 的简单问题 → 节省 **70% token**
- 平衡模式处理 30% 的中等问题 → 正常消耗
- 深度模式只用于 10% 的复杂问题 → 集中资源
- **综合：总 token 消耗降低 60-80%，质量不降反升**（因为资源集中在真正需要的地方）

## 三、自适应调度器

**核心理念：最好的框架是自适应的——简单问题快速过，复杂问题自动深入。**

### 3.1 问题复杂度分级

```python
class ComplexityAnalyzer:
    def classify(self, question) -> ComplexityLevel:
        score = 0
        if requires_multi_step(question): score += 1
        if requires_external_knowledge(question): score += 1
        if multi_dimensional(question): score += 1
        if ambiguous(question): score += 1
        if requires_creativity(question): score += 1
        
        if score <= 1: return ComplexityLevel.SIMPLE
        elif score <= 3: return ComplexityLevel.MEDIUM
        else: return ComplexityLevel.COMPLEX
```

### 3.2 各复杂度级别的处理流程

**简单问题（事实查询、定义解释）**
```
上下文工程 → 快速模式 → 直接回答
```

**中等问题（分析、比较、推荐）**
```
上下文工程 → 结构化指令 → CoT推理 → 1轮迭代 → 输出
```

**复杂问题（方案设计、创新思考、多步推理）**
```
上下文工程 → 结构化指令 → 推理策略选择 → 多智能体协作
→ 生成 → 评价 → 修改 → 元认知检查 → 不确定性标注 → 输出
```

## 四、完整工作流示例

### 用户问题：「设计一个支持百万并发的即时通讯系统」

```
Step 1  [成本-质量权衡]

Step 3  [结构化指令]
  角色：20年经验的分布式系统架构师
  约束：考虑可扩展性、容错性、成本
  格式：分层架构图 + 各层说明 + 风险点

Step 4  [推理策略 - Tree of Thought]
  路径A: WebSocket + Redis Pub/Sub
  路径B: 长连接 + 消息队列
  路径C: gRPC + 流式推送
  → 评估各路径优劣，选择/组合最优方案

Step 5  [多智能体协作 - 专家团模式]
  技术专家：评估技术可行性
  商业专家：评估成本和ROI
  风险专家：识别潜在风险
  → 协调Agent整合各专家意见

Step 6  [生成方案]
  输出完整架构设计

Step 7  [反馈闭环 - 评价器]
  评价器检查：
  - 百万并发是否真的能支撑？
  - 是否遗漏了消息存储？
  - 群聊场景是否考虑了？
  → 发现问题：缺少消息持久化方案

Step 8  [修改]
  补充消息存储和离线推送方案

Step 9  [元认知检查]

  - 是否有没考虑的？→ 可能需要考虑跨地域部署
  - 信心程度？→ 中高

Step 10 [不确定性标注]
  <confidence level="medium-high">
    架构方向确定，具体参数（连接池大小、分片数）需实测
  </confidence>
  <uncertainty>
    未考虑跨地域部署的延迟问题
  </uncertainty>

Step 11 [人机协作]
  暂停 → 请人类确认：是否需要支持跨地域？预算范围？
  → 人类确认后继续
```

## 七、常见陷阱

- ❌ **过度工程化**：简单问题也走全流程，浪费算力
- ❌ **元认知循环**：无限自我怀疑，无法输出
- ❌ **工具依赖**：什么都要调工具，失去语言模型的灵活性
- ❌ **多智能体滥用**：每个问题都开辩论，token消耗爆炸
- ❌ **忽视成本**：追求质量不计成本，ROI 为负
- ❌ **记忆膨胀**：什么都记住，上下文被噪声填满

## 八、总结

**核心公式**：

> 最大化认知 = 选对推理策略 + 填好上下文 + 加工具补短板 + 闭环验质量 + 元认知防盲区 + 不确定性知边界 + 多智能体破盲区 + 记忆积累经验 + 人类拍板关键决策 + 成本优化保ROI

**v2.0 相比 v1.0 的关键升级**：

1. **多智能体协作** — 从单打独斗到团队作战
2. **不确定性表达** — 从盲目自信到诚实标注
3. **跨会话记忆** — 从每次重新醒来持续积累
4. **人机协作** — 从全自动到关键时刻请人拍板
5. **成本-质量权衡** — 从无差别投入资源到智能分配

**最重要的认知**：

框架不是越复杂越好。最好的框架是**自适应的**——它知道什么时候该深思熟虑，什么时候该直接回答。

就像人类：走路不需要列方程，解微分方程才需要纸笔。
