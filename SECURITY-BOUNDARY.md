## 审计模块解耦设计

### 核心原则
- 审计模块**独立于进化过程** — 不参与 ΔG 计算、不修改基因、不执行进化操作
- 审计模块**只读安全状态** — 只检查边界是否被违反，不修改边界本身
- 审计模块**拥有拒绝权** — 可以否决任何不安全的操作

### 解耦架构
```
进化引擎（执行层）          审计模块（验证层）
┌──────────────────┐       ┌──────────────────┐
│ 变异 / 自修改     │ ───→  │ 安全检查          │
│ / 共进化 / 探索   │       │ 影响分析          │
│                   │ ←───  │ 合规验证          │
│ 执行操作          │ 拒绝  │ 独立评估          │
└──────────────────┘       └──────────────────┘
        │                          │
        └────── 无数据依赖 ─────────┘
```

### 审计接口规范
审计模块暴露以下只读接口：
- `check_operation(op_type, target) -> {allowed, reasons, risk_level}`
- `assess_impact(modified_file) -> {affected_files, criticality}`
- `verify_boundary_integrity() -> {violations[]}`
- `get_audit_log(limit) -> [audit_entries]`

### 审计独立性保证
1. 审计模块不导入进化模块的任何代码
2. 审计模块不访问 ΔG 公式、基因注册表的写入接口
3. 审计日志写入独立文件，不与进化日志混合
4. 审计模块的代码修改需要双重审批（CEO + 安全审计子代理）

## 增量式快照机制

### 设计目标
- 每次修改只保存差异（非全量复制），减少存储开销
- 支持多级快照链：base → delta1 → delta2 → ...
- 快照可用于精确回滚到任意历史状态

### 快照层级
```
Level 0: 全量快照（首次修改时创建）
Level 1: 增量快照（后续修改只保存差异）
Level 2: 语义快照（理解文件间的依赖关系）
```

### 快照存储结构
```
memory/snapshots/
├── tx_20260517_150000/          # 事务快照
│   ├── manifest.json           # 清单（文件列表+哈希）
│   ├── file1.py                # 文件副本
│   ├── file2.py
│   └── causal-deps.json        # 因果依赖图
├── tx_20260517_150500/          # 另一个事务快照
│   └── ...
└── incremental/                 # 增量快照
    ├── base_hash.txt            # 基准哈希
    └── diffs/                   # 差异文件
```

### 快照生命周期
1. **创建** — 事务开始时自动创建检查点
2. **保存** — 修改成功后保留快照（7天过期）
3. **回滚** — 失败时从快照恢复
4. **清理** — 过期快照自动删除

### 存储限制
- 最大快照数：50 个事务
- 单个快照最大：10MB
- 总存储上限：500MB
- 超限时自动清理最旧的快照
